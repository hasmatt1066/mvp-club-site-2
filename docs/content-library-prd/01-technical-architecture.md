# Technical Architecture — Content Library

## Current Site Architecture

**Stack:** React 18 + Vite 6 + react-router-dom v6 + Tailwind CSS, deployed on Vercel.

**Two rendering systems coexist:**

- **Main site pages** (/, /how-we-work, /for-organizations, /community, /assess, /case-studies/dual-enroll): Uses `vite-react-ssg` (v0.8.9) for static site generation. Routes defined in `/src/routes.jsx` with `React.lazy` imports. However, the current `dist/` output shows only a single `index.html` at the root — SSG may not be fully working as intended, with Vercel's SPA fallback handling these routes client-side.

- **Blog pages** (`/blog/*`): A completely separate, custom static site generator (`scripts/generate-blog.js`) reads JSON files from `content/blog/`, generates standalone HTML files into `public/blog/`. These are fully static, self-contained HTML pages with their own inline CSS, nav, and structured data — they share NO components with the React app.

**SEO tooling:**
- `src/components/SEO.jsx` uses `vite-react-ssg`'s `<Head>` component for meta tags, OG, Twitter cards, and JSON-LD on React pages
- Blog pages have their own hardcoded meta tags baked into HTML templates
- `sitemap.xml` generated by blog build script
- `robots.txt` allows all crawlers
- `llms.txt` exists for AI crawlers

**Content format:** Blog posts stored as JSON files in `content/blog/` with pre-rendered HTML in the `content` field. Also has Substack RSS feed integration. Currently 17 local blog posts.

## SPA SEO Challenges at Scale

The blog is already fine — static HTML, fully crawlable. But scaling to hundreds of pages introduces problems:

1. **Build time** — Current script is synchronous, rebuilds ALL pages every time. Should stay manageable even at 1000 pages since it's just file I/O.
2. **Content authoring in JSON** — Writing HTML inside JSON strings is painful. No syntax highlighting, error-prone. This is the biggest friction point.
3. **Design drift** — Blog pages are completely disconnected from React app styles/components. Any nav/brand/CTA change requires editing the blog template separately.
4. **No topic/category pages** — Current system has a single listing page. Hundreds of posts need category pages, topic clusters, pagination, related-post linking.
5. **Sitemap scaling** — Current flat sitemap works but should become a sitemap index at scale.

## Recommended Approach: Astro Migration

**Why Astro:**

1. **Content Collections** — Define a schema, write in Markdown/MDX, Astro handles validation, typing, and page generation. Adding a new article = create a `.md` file.
2. **Your React components work as-is** — `@astrojs/react` integration lets you use existing components. Add `client:load` or `client:visible` only to interactive ones.
3. **Build performance** — Benchmarked generating 10,000+ pages in under a minute.
4. **Zero-JS content pages** — Articles ship as pure HTML with zero JavaScript unless you add interactive components. Ideal for SEO and Core Web Vitals.
5. **Built-in integrations** — `@astrojs/sitemap`, `@astrojs/rss`, MDX support are all first-party.
6. **Vercel deployment** — First-party Vercel adapter. Same deployment process.

**Migration path:**

```
Phase 1: Scaffold Astro project, install @astrojs/react, @astrojs/tailwind
Phase 2: Move React components into Astro's component system
Phase 3: Convert existing JSON blog posts to Markdown (script-assisted)
Phase 4: Set up Content Collections with schema for articles
Phase 5: Build article template, listing page, category pages
Phase 6: Configure sitemap, RSS, structured data generation
Phase 7: Deploy to Vercel, verify everything works
```

**Alternative approach:** Enhance `generate-blog.js` to accept Markdown files instead of JSON, add `marked` as a dependency, add category/tag page generation, add pagination. Lower migration risk but accumulates maintenance burden.

## URL Structure Recommendations

```
/library/                              → Main library landing page (filterable)
/library/[article-slug]                → Individual articles (flat)
/library/topic/[topic-slug]            → Topic pillar pages
/library/author/[author-slug]          → Author pages (E-E-A-T)
/blog/                                 → Keep for timely/opinion content
```

**Rules:**
- **Flat article URLs** — Do NOT nest by date or category. Flat URLs avoid redirect issues when recategorizing.
- **Short, keyword-rich slugs** — Under 5-6 words
- **Topic cluster pages are critical** — Create pillar pages that link to all related articles
- **Author pages for E-E-A-T** — Bio, credentials, links to all their articles
- **Trailing slash consistency** — Match Vercel config (`trailingSlash: false`)

**Structured data for articles:**

Required fields: `headline`, `author` (Person), `datePublished`, `dateModified`, `publisher` (Organization), `description`, `mainEntityOfPage`

Add: `image`, `wordCount`, `BreadcrumbList` schema, `FAQPage` schema where applicable

**Sitemap at scale:**

```
/sitemap-index.xml
  /sitemap-pages.xml       → static pages
  /sitemap-library-0.xml   → library articles 1-500
  /sitemap-library-1.xml   → library articles 501-1000
```

Astro's `@astrojs/sitemap` handles this automatically.
